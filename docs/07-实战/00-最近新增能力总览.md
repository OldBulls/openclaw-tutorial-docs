---

## 新增了什么

最近新增内容围绕一个目标：让 OpenClaw 不只是“能聊天”，而是变成一个可以长期运行、能接企业 IM、能自检、能把记忆越用越干净的本地 AI 工作台。

| 能力 | 解决的问题 | 免费教程能做到什么 |
|---|---|---|
| 多模型中转与 fallback | 单一模型慢、贵、偶发不可用 | 配默认模型、兜底链、鉴权检查 |
| 多 agent 协作拓扑 | 一个 agent 什么都干，提示词越写越乱 | 拆 main / planner / executor / reflector |
| 共享记忆与周审 | agent 会忘、记忆文件越来越脏 | 建候选区、分层合并、定期回滚 |
| 飞书接入 | AI 只在终端里，团队用不起来 | 收消息、发卡片、群里协作 |
| 自定义交互卡片 | 只能“看报告”，不能点按钮执行 | 按钮携带 action，回调触发脚本 |
| 运维自检 | 升级、插件、权限一变就崩 | doctor、health check、smoke test |
| 安全交付 | 模板里容易混入个人密钥 | 占位符扫描、敏感文件隔离 |
| 安全更新链 | 买家端不知道该信哪份安装包 | `latest.json`、签名、公钥、阻断式更新 |
| gateway 路径自愈 | nvm / npm 路径漂移后 Control UI 打不开 | 检查 launchd / systemd 指向并修复 |
| housekeeper 巡检 | 定时任务和本地补丁容易静默失效 | 独立巡检脚本、心跳、运行时状态报告 |

这几类能力组合起来，就是一个可复制的闭环：

```mermaid
flowchart LR
    msg[飞书 / CLI 输入] --> main[main 统一入口]
    main --> planner[planner 拆任务]
    main --> worker[业务 agent 执行]
    worker --> mem[候选记忆]
    mem --> digest[周审卡片]
    digest --> action[用户点按钮]
    action --> merge[合并 / 回滚 / 重发]
    merge --> mem2[长期记忆]
    health[健康检查] -.守护.-> main
    health -.守护.-> digest
```

## 最近几版到底修了什么

如果你是从早期教程一路跟过来的，最近几版最重要的变化不是“又多了几个脚本”，而是把交付和运维闭环收紧了。

| 版本线 | 重点 | 对普通用户的意义 |
|---|---|---|
| `v1.0.6` | 把交付基线固定回 OpenClaw `2026.4.24` | 避免安装器把用户带到回复变慢或插件未验证的新 runtime |
| `v1.0.7` | memory-lancedb-pro 脚本兼容 nvm、系统 npm、`~/.local` | 买家不是和作者同一安装路径，也能打补丁和备份恢复 |
| `v1.0.8` | 买家在线更新拆成 `review` 和 `approve`，签名失败会阻断 | 不再只看版本号和 ZIP 哈希，先验证元数据来源 |
| `v1.0.9` | 新增 gateway 服务路径修复 | launchd / systemd 指向旧 Node 或旧 OpenClaw 时，可自动修回 |
| `v1.0.10` | gateway runtime 缺失时按基线自动补装，memory 补丁校验 15 秒默认值 | 全局 npm 包被清掉、`dist/index.js` 不存在时也能自愈 |
| `v1.0.11` | 作者端和买家端机器命令入口收敛 | OpenClaw / CLI 读 manifest 后可按关键词执行自动化 |
| `v1.0.12` | 周审源文件自动初始化，修复通用 🟢🟡🔴 分类漏扫 | 安装 / 更新后候选记忆和子 Agent 上卷建议链路更完整 |

这条线的共同目标是：**让模板不是只能在作者电脑上跑，而是换一台机器、换一种 Node 安装方式，也能通过预检、安装、更新、自检和支持包把问题收口。**

## 推荐搭建顺序

不要一上来就写复杂卡片。OpenClaw 的稳定性来自分层验证，每层都能单独回滚。

### 第 1 层：跑通本地 runtime

先完成：

- `openclaw setup`
- `openclaw gateway run`
- `openclaw chat main`
- `openclaw doctor`

对应章节：

- [安装](../01-入门/01-安装.md)
- [验证](../01-入门/03-验证.md)

判断标准：CLI 能和 main agent 正常对话，gateway 日志没有持续报错。

### 第 2 层：把“会话”变成“系统”

然后配置：

- `shared/`：团队规则、术语表、任务边界
- 多 agent：main 只路由，planner 只规划，业务 agent 只执行
- fallback models：主模型不可用时自动降级

对应章节：

- [基础配置](../01-入门/02-基础配置.md)
- [多 Agent 架构](../02-配置/00-多Agent架构.md)
- [协作拓扑](../03-多Agent/00-协作拓扑.md)

判断标准：你能明确说出“哪个 agent 负责什么”，而不是把所有规则塞进一个 prompt。

### 第 3 层：让记忆可审阅

接着做：

- 把原始记忆和候选记忆分开
- 按高价值、待审阅、低价值分层
- 每周生成 digest，由人确认再合并
- 合并后能回滚

对应章节：

- [记忆系统](../02-配置/01-记忆系统.md)
- [飞书周审卡片：从记忆候选到可点击审批](./02-飞书周审卡片.md)

判断标准：系统不会把所有聊天碎片都写进长期记忆，而是先进入候选区。

### 第 4 层：接入飞书，把 AI 放进工作流

飞书不要只当消息入口，更适合当“人机协作面板”：

- 文字消息：适合问答、派单、追问
- 卡片消息：适合报告、审批、二次确认
- 交互回调：适合触发合并、回滚、重发、打开配置引导

对应章节：

- [飞书深度配置](../02-配置/03-飞书深度配置.md)
- [飞书交互卡片入门](./01-飞书交互卡片入门.md)
- [飞书卡片回调处理器](./03-飞书卡片回调处理器.md)

判断标准：用户在飞书里点按钮，OpenClaw 能收到 action，并只执行白名单内的安全动作。

### 第 5 层：做运维闭环

最后补齐：

- gateway 健康检查
- gateway 服务路径修复
- 插件启停检查
- 日志轮转
- 卡片 dry-run
- handler 单元测试
- 升级前备份和升级后 smoke test
- housekeeper 巡检和心跳任务

对应章节：

- [健康检查](../04-运维/00-健康检查.md)
- [日志与监控](../04-运维/01-日志与监控.md)
- [CLI 与 Gateway 版本统一](../06-升级/03-CLI与Gateway版本统一.md)
- [版本统一与脚本治理实战](./12-版本统一与脚本治理实战.md)
- [飞书卡片验证与排障](./04-飞书卡片验证与排障.md)

判断标准：你不需要等用户投诉才知道系统坏了。

## 免费文档和付费模板的边界

免费文档会尽量讲清楚“为什么这样设计”和“怎么自己做出来”：

- 架构拆分方法
- 配置位置和验证命令
- 飞书卡片的数据流
- handler 的安全边界
- 常见坑和排障路径
- 安装包签名校验和更新阻断原则
- CLI / gateway 版本统一和路径漂移排障

付费模板主要节省时间：

- 去个性化后的配置骨架
- 可直接改占位符的脚本模板
- 安装器和 smoke test
- 在线安全更新和签名校验脚本
- 多 agent shared 规范模板
- 周审卡片、合并、回滚的完整实现骨架
- housekeeper 巡检、gateway 修复、memory-lancedb-pro 本地补丁模板

如果你愿意自己写代码，免费文档足够搭出同类系统；如果你想少走弯路，模板是加速器。

## 最小可复刻清单

想跟着 AI 一步步做，建议把任务拆成 6 个 prompt，不要一次让 AI 生成全部：

1. “根据我的业务，帮我设计 OpenClaw 的 main / planner / worker / reflector 职责边界。”
2. “帮我写 shared 目录的规则文件，要求可被多个 agent 读取。”
3. “帮我设计一个记忆候选 Markdown 格式，分高价值、待审阅、低价值三层。”
4. “帮我生成一张飞书 CardKit 2.0 审阅卡片，只包含统计、明细折叠区和三个按钮。”
5. “帮我写一个只接受白名单 action 的回调 handler，按钮 value 里必须带 task_id 和 issued_at。”
6. “帮我写 dry-run、日志和回滚流程，先不接真实飞书 API。”

每一步都先在本地 dry-run，再接真实飞书。这样最稳。
