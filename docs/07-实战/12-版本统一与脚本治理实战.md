> 目标：用一组真实工程场景把三件事串起来：**CLI / gateway 版本统一、memory 迁移、自定义脚本治理**。这不是理论课，而是一条能直接套用到你自己环境里的收口路径。

---

## 这篇适合什么时候读

如果你已经遇到下面任意一个症状，直接读这一篇：

- `config validate` 说某个键非法，但 gateway 明明又要求你设置它
- embedding 维度从 1024 切到 4096 后，记忆系统起不来
- 你自己的 `~/.openclaw/scripts/` 开始“帮倒忙”
- 飞书和 WebUI 一起不回，但 gateway 进程看起来还活着

---

## 一条真实的故障链长什么样

这类系统性事故通常不是单点，而是链式叠加：

1. **模型 / 中转站配置收敛**
   你刚刚统一了主模型、fallback、压缩模型。
2. **embedding 升级到 4096 维**
   配置是新的，LanceDB 表还是旧的。
3. **自定义自愈脚本回滚 baseline**
   刚改好的 `dbPath`、rerank 配置又被抹回旧值。
4. **后台 review 继续抢主会话**
   `agent:main:main` 卡死，前台开始一起不回。
5. **CLI 和 gateway 版本还不一致**
   你看到的 warning 和你能不能写配置键再次打架。

这时候如果你分开一个个猜，会很慢。  
正确策略是按“依赖关系”收：

```text
版本统一
  ↓
配置定稿
  ↓
迁移记忆库
  ↓
治理自定义脚本
  ↓
清理坏会话 / 背景任务噪音
```

## 第一层：先统一版本，不然你连“什么算正确配置”都说不清

先看：

```bash
which -a openclaw
openclaw --version
openclaw gateway status
```

如果 CLI 和 gateway 不是同一版，优先统一。  
详见 [06-升级/03-CLI与Gateway版本统一](../06-升级/03-CLI与Gateway版本统一.md)。

**原则**：

- 先统一入口
- 再加新配置键
- 最后再重启服务

---

## 第二层：embedding 升级时，新库重建比“原地硬改”更靠谱

你要升到：

```text
Qwen/Qwen3-Embedding-8B + 4096
```

不要只改：

- `embedding.model`
- `embedding.dimensions`

而是：

1. 新建 `dbPath`
2. `reembed`
3. 校验计数
4. 再切流量

详见 [04-运维/07-LanceDB-4096维迁移与重嵌入](../04-运维/07-LanceDB-4096维迁移与重嵌入.md)。

---

## 第三层：脚本只补缺省，不要整段回滚

真实环境里最危险的一类脚本不是“完全坏掉”，而是“看起来在修复，实际上在覆盖生产新配置”。

最典型的错误就是：

```text
baselineMemory != currentMemory
→ 全部恢复成 baseline
```

正确做法：

```text
只补 baseline 里有、当前缺失的字段
保留已经显式改过的生产配置
```

详见 [04-运维/08-自定义脚本与Cron治理](../04-运维/08-自定义脚本与Cron治理.md)。

---

## 第四层：后台任务必须和主会话隔离

当你开始做：

- background review
- 周审
- 自动归并
- 自愈脚本

这些后台动作时，默认不要去抢 `agent:main:main`。

应该做的是：

- 最近活跃会话门槛
- `pending` / `running` 哨兵
- 独立 `session-id`
- 快速 `/health` 门槛
- 派发硬超时

这样就算后台任务失败，也只是“没跑成”，不会顺手把主控拖死。

详见 [05-排障/02-主会话锁死与后台任务噪音](../05-排障/02-主会话锁死与后台任务噪音.md)。

## 第五层：把“修完了”变成可复用经验

很多人修完就结束，下一次同类事故还从头踩。

更好的做法是把经验拆成三层：

### 层 1：当天日志

记录：

- 现象
- 根因
- 怎么修
- 哪些命令有效

### 层 2：durable correction

只保留真正可复用的原则，比如：

- CLI 和 gateway 必须版本统一
- 自愈脚本只补缺省，不整段回滚
- 4096 维迁移必须新库重嵌入

### 层 3：公开教程

把“这不是你一个人的坑”的部分写成文档。  
这样以后不需要从聊天记录里翻答案。

---

## 这套收口流程的价值

它不是“修好一个 bug”，而是把系统稳定性往上抬一层：

- 版本口径统一
- 数据口径统一
- 脚本边界清晰
- 后台任务不再反噬主控
- 同类问题可文档化复用

这才是长期运行 OpenClaw 最值得做的那类工程工作。

---

## 下一步

如果你现在已经处在“能跑，但怕再炸”的阶段，建议按这个顺序继续：

1. 把所有高风险脚本做一次 owner 清单
2. 把 memory / model / channel 的基线和生产配置分开
3. 给每条后台任务加 lock、cooldown、独立 session id
4. 把已经验证过的经验写进公开文档，而不是只留在聊天记录里
