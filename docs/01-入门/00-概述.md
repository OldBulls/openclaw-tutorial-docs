## 一句话定位

**OpenClaw 是一个本地运行的多 AI Agent 运行时平台。**

它让你在一台机器上跑若干个各司其职的 AI Agent，通过飞书/Discord 等渠道与用户连接，带长期记忆、带协作规则、带自愈运维。

---

## 和"直接用 ChatGPT / Claude API"的区别

| 直接调 API | OpenClaw |
|---|---|
| 每次请求独立，无记忆 | 五层记忆架构（L0~L4），跨会话持久 |
| 单个 agent，所有事一锅煮 | 多 agent 分工，按职责路由 |
| 渠道接入要自己写 webhook | 飞书集成开箱可用 |
| 配置改动即生效，错了自担 | `openclaw doctor` 自检 + 升级脚本 |
| Prompt 散落在代码里 | SOUL / AGENTS / TOOLS 规范化 |
| 没有任务追踪 | `taskflow` 内置任务状态机 |

如果你要做的只是"一次性调用 LLM 返回 JSON"，OpenClaw 是过度工程。
如果你要做的是"一个能**持续在线、记得你、会分工**的 AI 系统"，它就是合适的地基。

## 核心架构（从下往上）

```
┌────────────────────────────────────┐
│  渠道层（Channel）                   │
│  飞书（主） / Discord / Telegram     │
├────────────────────────────────────┤
│  Gateway                            │
│  port 18789 · HTTP + WebSocket       │
│  路由消息 → 对应 agent → 回传         │
├────────────────────────────────────┤
│  Agent 层                           │
│  main / planner / 业务agent / 专项   │
│  通过 sessions_send / spawn 互相协作  │
├────────────────────────────────────┤
│  Skills / Plugins 层                 │
│  lossless-claw · proactivity        │
│  memory-lancedb-pro · self-improving │
│  ontology · delegate-task · ...     │
├────────────────────────────────────┤
│  记忆层（5 层）                      │
│  L0 lossless-claw（会话内 DAG 压缩）  │
│  L1 proactivity（运行态推进）         │
│  L2 memory-lancedb-pro（语义向量）    │
│  L3 self-improving + memory/（经验）  │
│  L4 Graphify（知识图谱）              │
├────────────────────────────────────┤
│  配置层                              │
│  ~/.openclaw/openclaw.json          │
│  ~/.openclaw/shared/ (全局规范)      │
│  ~/.openclaw/agents/*/  (每个agent)  │
│  ~/.openclaw/credentials/ (密钥)     │
└────────────────────────────────────┘
```

## 关键术语

| 术语 | 含义 | 典型例子 |
|---|---|---|
| **Agent** | 一个 AI 实例，有独立的 SOUL（人设） / TOOLS（能力） / 工作空间 | `main` / `planner` / `minsu` |
| **Session** | agent 的一次运行实例，支持多轮 | 一条与 main 的飞书对话 |
| **Gateway** | 核心服务，管理所有 agent 运行时 | `openclaw gateway run`（前台）/ `openclaw gateway start`（launchd 托管） |
| **Skill** | 可插拔能力模块，像 App | `memory-lancedb-pro` 提供语义召回 |
| **Tool** | agent 可调用的具体工具 | `feishu_im_user_message send` |
| **sessions_send** | 向**已有** session 发消息并等回复 | 继续已有对话 |
| **sessions_spawn** | **新建** session（`mode=run` 一次性 / `mode=session` 持续） | 起新任务 |
| **SOUL.md** | agent 的人设 / 核心原则 | `agents/main/SOUL.md` |
| **AGENTS.md** | 协作规则 / 派发决策树 | `shared/AGENTS.md` |
| **TOOLS.md** | 工具清单 / 执行规范 | `shared/TOOLS.md` |
| **bootstrap-extra-files** | 把 `shared/` 下的规范自动注入每个 agent 的启动上下文 | 无需手动同步 |

## 五层记忆（一张表理解）

| 层 | 实现 | 解决什么问题 | 存哪 |
|---|---|---|---|
| L0 | lossless-claw | 单次对话 context 爆了 | 内存（会话内） + `~/.openclaw/lcm-files/` 压缩快照 |
| L1 | proactivity | 当前任务推进到哪 | `~/.openclaw/proactivity/memory.md`（全局共享，非 per-agent） |
| L2 | memory-lancedb-pro | 跨会话事实召回 | `~/.openclaw/memory/lancedb-pro/memories.lance/` |
| L3 | self-improving + `memory/` | 稳定规则 / 日期日志 | `~/self-improving/` + `~/.openclaw/workspace/memory/YYYY-MM-DD-{话题}.md`（扁平日期命名） |
| L4 | Graphify | 实体/关系图谱 | `~/.openclaw/data/graphify/openclaw-memory/graphify-out/` |

**不要混层级用**：当天的工程结论写 L3 的 `memory/YYYY-MM-DD.md`，稳定规则进 `MEMORY.md`，语义召回靠 L2。详见 [02-配置/01-记忆系统](../02-配置/01-记忆系统.md)。

## 什么样的用户适合

**适合**：

- 需要**长期在线**的 AI 助手（飞书客服、私域运营、团队内部自动化）
- 需要**多 agent 分工**（一个 main 入口 + 若干业务 agent）
- 对"agent 记得我"有刚需（而不是每次重新介绍自己）
- 愿意花一天配置，换未来半年稳定运行

**不适合**：

- 只要一次性调用（用 API SDK 更快）
- 没有飞书/Discord 之类的交互渠道诉求
- 不能接受命令行 / JSON 配置 / 日志排障

## 一个具体的场景

以作者的真实部署为例：

- **main**（家族大脑）—— 飞书私聊入口，负责总协调
- **planner**（规划师）—— 任务拆解、SOP 设计
- **minsu**（民宿管家）—— 民宿日常问答、上房流程
- **yying**（运营助手）—— 小红书内容制作
- **zhaosheng**（招生助理）—— 小学招生宣传
- **reflector**（反思者）—— 周期性整理记忆、输出审阅卡

用户在飞书发"这周民宿收益怎么样"，main 路由到 minsu，minsu 查 bitable 汇报。
用户发"规划下周的招生活动"，main 路由到 planner 拆解，再派给 zhaosheng 执行。

这种"按职责分工 + 飞书统一入口"是 OpenClaw 最常见的用法。
