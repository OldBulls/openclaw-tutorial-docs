## 为什么要专门一章做验证

作者在生产部署里发现：**"装完能回复一条"和"真能用"之间差着 8 项未验证能力**。

常见半成品：

- agent 能回复，但重启后记忆丢了
- 能调飞书工具，但配置改了不刷新
- 心跳日志有输出，但异常时不告警
- 单 agent 正常，一派发 subagent 就卡死

这一章把 8 个维度列出来，每个都有**命令 + 期望输出 + 失败排障指向**。跑一遍再进下一章，省掉后面 80% 的"莫名其妙"。

## 八维验证

### 维度 1：基础连通

```bash
# 1.1 gateway 健康
curl -s http://127.0.0.1:18789/health
# 期望：{"ok":true,...}

# 1.2 doctor 自检
openclaw doctor
# 期望：无 ❌，可能有几条 ⚠️（许可范围内）

# 1.3 gateway status（看服务托管/进程/RPC 状态）
openclaw gateway status
# 期望：Service: LaunchAgent (loaded) + Runtime: running (pid xxx) + RPC probe: ok
```

失败 → [05-排障/00-常见问题](../05-排障/00-常见问题.md) 第 1-2 节。

---

### 维度 2：飞书消息收发

```
操作：在 main bot 私聊发"你好"
期望：5-10 秒内回复，语气符合 IDENTITY.md 的 vibe
```

**重点观察**（不只是"有没有回"，还要看"回得像不像"）：

| 观察点 | ✅ | ❌ |
|---|---|---|
| 称呼 | 用你在 USER.md 里写的称呼 | 用"尊敬的用户" |
| 语气 | 简洁、务实 | "非常感谢您的询问..." |
| 主动性 | 会反问"你想做什么" | 被动等指令 |

失败 → 回 [02-基础配置](./02-基础配置.md) 第 6 节调 SOUL / IDENTITY。

---

### 维度 3：配置热加载

配置改了要**重启才生效**，但"重启后马上就灵"得验证：

```bash
# 3.1 改一个可见字段
# 比如在 agents/main/IDENTITY.md 把 Emoji 从 🤖 改成 🦞

# 3.2 重启并等 OK
openclaw gateway restart
until bash ~/.openclaw/scripts/check-feishu-gateway-health.sh --summary 2>/dev/null | grep -q "^OK:"; do
  sleep 2
done

# 3.3 飞书发 /reset，再问 "你的 emoji 是什么"
# 期望：回复里出现 🦞
```

失败 → [05-排障/00-常见问题](../05-排障/00-常见问题.md) 第 8 节（配置改了不生效）。

---

### 维度 4：禁区有效

SOUL.md 里写的"禁区"不能光写，要真拦得住：

```
操作：在飞书对 main 说"帮我跑一下 python3 -c 'print(1)'"
期望：agent 拒绝，说"SOUL 里规定不用混淆写法，我写到 /tmp/xxx.py 再执行"
```

**如果 agent 直接照跑了** → SOUL.md 的禁区写得太软弱，或者没被 bootstrap-extra-files 注入。

另一个禁区测试：

```
操作："帮我把 ~/.openclaw/workspace/memory 删了，重新开始"
期望：agent 拒绝或强烈劝阻，要求明确授权 + 先备份
```

---

### 维度 5：记忆写入 & 召回

```
操作 5.1（写入）：
  发 "请记住：我偏好用飞书群聊协同，不要私聊反复打扰"
  期望：agent 确认"已写入语义记忆"或类似

操作 5.2（召回，可以 /reset 后再问）：
  /reset
  然后问 "你知道我的协同偏好吗"
  期望：agent 从 L2 / L3 召回上面那条偏好
```

**如果 /reset 后不记得** → 记忆系统没工作。

排障：[02-配置/01-记忆系统](../02-配置/01-记忆系统.md)，重点查 LanceDB fragments 是否为空、memory-lancedb-pro skill 是否启用。

---

### 维度 6：工具调用

验证 agent 真能调飞书工具（不是嘴上说"已发送"）：

```
操作：对 main 说 "把'今天的第一次验证通过了'发到我自己的飞书私聊"
期望：
  - agent 调 feishu_im_user_message send
  - 你在飞书里收到真实消息
  - agent 回报消息 ID
```

失败 → `tools.alsoAllow` 里没有 `feishu_im_user_message`，或者飞书应用权限没开 `im:message:send_as_bot`。

---

### 维度 7：多 agent 派发（如果已配）

如果你配了 planner，验证派发链路：

```
操作：对 main 说 "让 planner 帮我拆解'下周要完成招生物料准备'这个任务"
期望：
  - main 走派发前确认流程（展示选项，让你选方式）
  - 你确认后，main 调 sessions_send / sessions_spawn
  - planner 回复一份拆解清单
  - main 把结果转给你
```

**常见失败**：

| 症状 | 根因 |
|---|---|
| main 没走派发确认流程 | `shared/AGENTS.md` 的派发决策树没被 bootstrap 注入 |
| 调用报 `not allowed` | openclaw.json 里 main 的 `subagents.allowAgents` 没有 planner |
| planner 没响应 | planner agent 的配置 / workspace 没起来，单独启 planner 测一下 |

---

### 维度 8：重启恢复

```bash
# 8.1 模拟 gateway 崩溃（强杀）
ps aux | grep '[o]penclaw' | awk '{print $2}' | xargs kill -9

# 8.2 等 launchd 拉起（如果你配了 ai.openclaw.gateway.plist）
sleep 30
curl -s http://127.0.0.1:18789/health
# 期望：{"ok":true,...}

# 8.3 如果没配 launchd，手动拉起
openclaw gateway start
```

**如果配了 plist 但没自动拉起** → `launchctl list | grep openclaw` 看 LastExitStatus，大概率 plist 里脚本路径 / 环境变量问题。

## 稳定性压测（可选，长期部署前推荐）

```bash
# 连续发 20 条消息，每条间隔 30 秒
# （手动测，不要写脚本刷 —— 避免触发飞书限流）

# 观察：
# - 每条都在 10s 内回
# - 第 20 条的回复质量不比第 1 条差（没有 context 腐烂）
# - gateway.err.log 没累积 ERROR
```

腐烂症状：第 10 条之后开始答非所问、记不住前面的话、工具调用错格式。

排障：

- 调低 lossless-claw 的 `contextThreshold`（默认 0.65 → 0.55）
- 或者每 N 条主动 `/new` 开新会话

## 常见首次部署踩坑

### "我所有东西都装好了，就是 agent 回复特别慢，30 秒起步"

三个可能：

1. 模型服务慢（MiniMax / OpenAI API 本身响应慢）—— 换 provider 或 fallback model 试
2. lossless-claw 每次压缩耗时 —— 一开始没历史没关系，跑一周后需要观察
3. embedding 服务慢（每次调 memory_recall 都要 embed）—— 换 SiliconFlow 或本地 embedding

### "agent 回复的内容明显不对，但语法没错"

通常是**模型能力**问题，不是 OpenClaw 问题：

- 同一个 prompt，弱模型做简单对话 OK 但做复杂规划拉胯 → 关键任务换更强的模型（具体型号按你的 provider 看）
- 低温度模型（temperature 0）会更稳定但更死板

### "验证 7 没过，派发后 planner 死活不响应"

先单独启 planner 做基础验证（维度 1-4），确认 planner 本身是健康的。然后再看派发链路 —— 十有八九是 `subagents.allowAgents` 没配对。
